## Priority 1 — Bugs / Blockers

### [STORY-15] Fix hardcoded credential defaults in config.py ⛔ BLOCKS PR #8
**As a** security engineer, **I want** `SECRET_KEY`, `admin_username`, and `admin_password` to raise a startup error if not supplied via environment **so that** no deployment can run with well-known credential defaults.  
**Severity:** CRITICAL — hardcoded `SECRET_KEY` allows forging valid JWTs; default `admin/changeme` enables trivial admin access.  
**References:** GitHub security issue #10. CWE-798 (Hard-coded Credentials).  
**Acceptance criteria:**
- Given `SECRET_KEY` is not set in the environment, When the app starts, Then a `ValueError` is raised immediately and the process exits — the app does NOT start
- Given `ADMIN_USERNAME` or `ADMIN_PASSWORD` is not set in the environment, When the app starts, Then the process exits with a clear error message naming the missing variable
- Given `config.py` is inspected in HEAD, When any credential-like literal is searched (e.g. "changeme", "dev-secret-key", "supersecret"), Then no such string appears as a Pydantic field default
- Given `SECRET_KEY` is set via env var to a value ≥32 characters, When the app starts, Then the JWT signing key is loaded from the environment and the app starts normally
- Given `POST /auth/token` is called with credentials matching the environment vars, When the request is processed, Then a valid JWT is returned  
**Owner:** Engineer  
**Estimate:** XS  
**GitHub Issue:** #17 (security finding: #10)  
**Resolve in:** PR #8 (must be fixed before STORY-13 merges)

### [STORY-16] Fix plain-text admin password comparison — use bcrypt ⛔ BLOCKS PR #8
**As a** security engineer, **I want** `POST /auth/token` to verify the admin password using bcrypt constant-time comparison **so that** the password is never compared as plain text and timing attacks are mitigated.  
**Severity:** HIGH — plain-text string equality is vulnerable to timing attacks and contradicts the stated design (`CLAUDE.md`: "Login password hashed at startup"). `bcrypt` is imported in `auth.py` but unused.  
**References:** GitHub security issue #13.  
**Acceptance criteria:**
- Given the auth module is inspected, When the password verification code is read, Then `_pwd_context.verify(form_data.password, hashed)` or `bcrypt.checkpw(...)` is called — NOT a `==` string equality comparison
- Given the password is hashed at app startup (not at request time), When `POST /auth/token` is called with the correct password, Then bcrypt verification passes and a JWT is returned
- Given `POST /auth/token` is called with an incorrect password, When bcrypt verification runs, Then HTTP 401 is returned
- Given the test suite runs, When auth tests execute, Then no test fixture or mock stores the admin password as a plain string in an assertion or log line
- ⚠️ The admin password must never appear in log output at any log level  
**Owner:** Engineer  
**Estimate:** XS  
**GitHub Issue:** #18 (security finding: #13)  
**Resolve in:** PR #8 (must be fixed before STORY-13 merges)

### [STORY-17] Fix python-jose CVEs — upgrade to ≥3.4.0 or migrate to PyJWT ⛔ BLOCKS PR #8
**As a** developer, **I want** JWT signing/verification to use a library without known algorithm-confusion CVEs **so that** the authentication guarantee is not undermined by a vulnerable dependency.  
**Severity:** HIGH — `python-jose==3.3.0` has 3 known CVEs (PYSEC-2024-232, PYSEC-2024-233, PYSEC-2025-185). Algorithm confusion attacks can allow an attacker to forge tokens that pass signature verification.  
**References:** GitHub security issue #11.  
**Recommended paths:**
- Minimal change: pin `python-jose[cryptography]>=3.4.0`
- Preferred (actively maintained): migrate to `PyJWT>=2.8.0`  
**Acceptance criteria:**
- Given `requirements.txt` is updated, When `pip-audit -r requirements.txt` runs, Then no CVEs are reported for the JWT library
- Given a JWT is generated by `POST /auth/token`, When the token is presented to a protected route, Then the token is decoded and the request proceeds normally
- Given an expired JWT is presented to a protected route, When the route handler runs, Then HTTP 401 is returned
- Given a tampered or invalid JWT is presented, When a protected route is called, Then HTTP 401 is returned
- Given the updated library is installed, When `pytest tests/ -v` runs, Then all existing JWT-related tests pass  
**Owner:** Engineer  
**Estimate:** XS  
**GitHub Issue:** #19 (security finding: #11)  
**Resolve in:** PR #8 (must be fixed before STORY-13 merges)

### [STORY-19] Migrate CI credential-shaped env vars to GitHub Actions secrets pattern ⛔ BLOCKS PR #9
**As a** security engineer, **I want** the CI workflow to reference all credential-shaped variables via `${{ secrets.VAR || 'fallback' }}` **so that** no developer is normalised into committing credential values in version-controlled YAML.  
**Severity:** LOW — current values are test placeholders, not real credentials. Risk is pattern normalisation: a developer who later replaces a placeholder with a real value will commit it to git history.  
**References:** GitHub security issue #16.  
**Acceptance criteria:**
- Given the CI YAML is inspected, When credential-shaped env vars are reviewed (`SECRET_KEY`, `ADMIN_PASSWORD`, `BETFAIR_USERNAME`, `BETFAIR_PASSWORD`, `BETFAIR_APP_KEY`), Then no plaintext credential values appear; all use `${{ secrets.VAR_NAME || 'safe-fallback' }}` form
- Given the CI workflow runs on a PR with no repository secrets configured, When tests execute with fallback values, Then unit tests pass (all Betfair and Odds API calls are mocked — no real API calls in tests)
- Given a developer reads the YAML, When they see the credential env vars, Then the pattern clearly communicates "inject via secrets, do not hardcode"  
**Owner:** DevOps  
**Estimate:** XS  
**GitHub Issue:** #21 (security finding: #16)  
**Resolve in:** PR #9 (must be fixed before STORY-1 merges)

### [STORY-7] Add rate limit guard to OddsApiService
**As a** developer, **I want** the OddsApiService to halt polling when quota is low **so that** the 500 req/month hard limit is never breached.  
**⚠️ The Odds API limit is 500 req/month — this guard is mandatory before enabling the scheduler.**  
**Acceptance criteria:**
- Given `x-requests-remaining < 50`, When the APScheduler job calls `OddsApiService`, Then the HTTP request is NOT made and a `WARNING` is logged with the current remaining count
- Given `x-requests-remaining >= 50`, When the scheduler calls `OddsApiService`, Then the HTTP request proceeds normally
- Given the guard has fired at least once in this process lifetime, When `GET /odds/api-status` is called, Then the response includes `{"requests_remaining": <n>, "guard_active": true}`
- Given the guard has never fired, When `GET /odds/api-status` is called, Then the response includes `{"requests_remaining": <n>, "guard_active": false}`
**Owner:** Engineer  
**Estimate:** S  
**Depends on:** STORY-3
